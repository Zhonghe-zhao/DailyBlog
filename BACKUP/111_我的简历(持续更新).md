**工作一: Coogo 云原生安全执行框架轻量化重构｜Go**

参与 Coogo 云原生安全执行框架拆分，基于原 Module 接口中的 CanI() 检测链路与 Run() 利用链路，将单体框架拆分为 poc-shell-core 与 exp-shell-core 两类执行核心，分别承载轻量扫描与利用执行能力。

分析 Go package 编译机制、init() 自注册模式与模块依赖树，定位 modules/escape 全量导入导致扫描产物携带 Kubernetes SDK、containerd、embed payload 等重依赖的问题；通过独立 Scanner 入口、Build Tags、Stub 替身与公共依赖轻量化改造，产出约 2.5MB 的 PoC 扫描核心。

基于公司 Coogo 能力表设计扫描结果优先级模型，将能力划分为 T0 配置类、T1 组件类、T2 内核类与 Info 类，并结合 CanI() 返回状态实现分层排序，优化原 Yes/Unknown/No 无序输出，提升靶场环境下漏洞验证路径选择效率。

**工作二：AI Agent 靶场能力接入与调用链路封装｜Go / Qwen-Agent / Gradio**

参与云 AI 攻防靶场中 Qwen-Agent Code Interpreter 能力接入，梳理 LLM 服务、Gradio 前端、Agent 执行逻辑与 Code Interpreter 后端/沙箱之间的调用边界。

通过抓包分析 Gradio WebUI 工具调用流程，明确 predict、queue/join、session_hash、fn_index、history 与流式返回数据结构，并在 Coogo 中新增 exec-qwen-agent 模块，将原本依赖页面交互的 Agent 执行能力封装为 CLI 可调用能力。

基于 Go 实现 Qwen-Agent headless 调用链路，构造 Gradio 请求 payload，兼容不同接口路径，解析 SSE/队列响应中的 process_generating 与 process_completed 状态，并从嵌套 JSON 与 Observation 输出中提取执行结果。

在本地与靶场环境中验证 Agent 工具调用稳定性，排查模型安全策略、后端沙箱限制、网络环境与输出格式差异对执行结果的影响，为云 AI 靶场的自动化能力调用提供可复现链路。
